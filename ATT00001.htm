In this image, AS55 is the one we are interested in. The large red circles on the edge are the RIPE data collectors and the numbers in the middle are all the AS that provide transit for traffic from the collector points to AS55.

Nice to haves here:
  • Animate the image so you can see it change over time
  • See somewhere which networks are hosted by AS55.
  • Mouse over a data collector:
    • highlights all routes that were seen from that point.
    • see a list of networks that are hosted by AS55 but are not announced to that data collector (so you can see outages)

The circular layout is an option, but perhaps there is a better (or easier) way.

The Data
The idea is to create the visualization / UI completely in HTML5 / JavaScript. Luckily, at RIPE, there is a server that will give the data required for this diagram in JSON. I will provide access to that server through a state-of-the-art network facility called a SSH tunnel.

The server is quite easy: you POST a query in JSON format in the request body and you get results back in JSON format. The query syntax is a bit involved, but here is the query that you will need:

 {
    "select": ["META_DATA", "BLOB", "TIMELINE"],
    "data_class": "RIS_RIB",
    "where": {
        "composite": {
            "primary": {
                "time_index": {
                    "from": "18:38:26.58 08-12-2010",
                    "to": "18:38:26.58 09-12-2010"
                }
            },
            "secondary": {
                "asn": {
                    "match": "EXACT",
                    "exact_match": "INCLUDE_EXACT",
                    "position": 0,
                    "direction": "FROM_RIGHT",
                    "resource": "AS3333"
                }
            }
        }
    },
    "and": [],
    "time_criterion": {
        "between": {
            "minimum_start_time": "18:38:26.58 08-12-2010",
            "maximum_end_time": "18:38:26.58 09-12-2010"
        }
    },
    "combine_timeline": true
}

The important parts are bold again. You need to pass in the start and end date twice (it’s an open issue, so for now we’ll call it a feature) and the AS number that you are looking for (AS3333 belongs to RIPE NCC itself). This query will ask for all the records in a given time frame where AS3333 is the rightmost AS number in the AS path (hence, it will give only records for networks which are hosted by AS3333).

Some explanation:

attribute
what is it
data_source
The name of the data collection point. There are 17 of these, named rrc00 to rrc16. Some of them don’t work (I think 2, 7 and 8), so you will never see data for those.
intervals
Series of time intervals denoted by a start and stop time in millis since epoch. The data of the record is said to be true during the given intervals. So in the las record, rrc00 was receiving announcements for 193.0.10.0/23 during the given interval. If you query for a short period of time, you will usually only see one interval for each record, but there could be more. Intervals will never overlap or be directly adjacent (two intervals for the same record with no time in between will be combined on the server side).
prefix
The network that can be reached through the announced route. So everything within 193.0.10.0/23 is hosted by AS333 and can be reached from rrc00 through 2 different AS paths (look at the first and the last record in this result).
as_path
The path of AS involved in the route. You should expect duplicates here as well as AS paths which have the same AS in them a number of times (e.g. 1234 5678 5678 5678 5678 3333 is valid). The people running the internet apply strange hacks all the time. The AS paths with the same number repeated a couple of times is used to make a route less favorable by making it artificially longer, such that routers will not give it priority.


In reality you should expect more than just 3 records. If you actually do the query above, you will get 688 records in 250K of JSON, which is one day worth of data.

To use the server, you need to post to this URL: http://<hostname>:61080/inrdb/query. I don’t know the hostname yet, but I will set it up somewhere and communicate that. It will probably be my laptop with a tunnel to the actual machine at RIPE. Because things will go through a tunnel and probably over WiFi, it might not be as snappy as you’d like. The above query took about 1.5 seconds in such a setup. Also, some data is really rarely requested so caches will be cold and the database needs to do a lookup amongst quite a substantial amount of this data, so the first time you do a query it could take seconds before it responds. The next time you do the same query will then come from cache and be fast.


The WebSocket Bonus
If you want to make the diagram animate over time, you will need more data than just a snapshot. As mentioned above, the data is about 250K per day, so drawing a month or scrolling through years is not really an option. In order to make this possible, you could write a server side component to do the queries for you and push the results to the client as they arrive. You could even strip the results of the unwanted fields. The client should be able to request different time windows through the same channel.

If you like the WebSocket bonus more than the visualization part, you can of course opt to start with that and just display the data textually or some other less complex way.
